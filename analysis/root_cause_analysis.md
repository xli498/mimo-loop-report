# MiMo Degenerate Loop — Root Cause Analysis

## Executive Summary

使用 `xiaomimimo/mimo-v2.5-pro` 模型时，当 `reasoning=True`，模型有约 30% 的概率进入退化循环。本文档基于实际观察数据，提供详细的根因分析和修复方案评估。

## 观察数据

### 退化循环特征

| 指标 | 数值 |
|------|------|
| 触发概率 | ~30% |
| 重复次数 | 8-11 次 |
| 总持续时间 | ~6 分钟 |
| 每次重复间隔 | ~33 秒 |
| 输出语言 | 英文（无视中文 System Prompt） |
| 工具调用 | 0 次 |
| frequency_penalty | 1.0（已启用，无效） |
| temperature | 0.7 |

### 语言切换模式

```
Block 1: 中文，正常输出（理解任务，描述计划）
Block 2: 中文输出 → 尝试调用工具（但模型端未执行）
Block 3-11: 英文输出，完全相同内容，循环重复
```

从 Block 3 开始，模型完全切换为英文，且无视 System Prompt 中的中文指令。
这表明模型的推理路径从"遵循指令的多语言路径"切换为"固定的英文推理路径"。

## 根因分析

### 核心假设：Reasoning=True 导致 Token 概率分布塌缩

MiMo 模型的 `reasoning=True` 模式内部工作流程：

```
User Input → Reasoning Engine (English-only) → Output Decoder → Response
```

**问题机制**：

1. **推理引擎激活**：`reasoning=True` 激活了模型内部的推理引擎，该引擎以英文为核心思考语言

2. **路径塌缩**：在某些输入条件下（特别是系统检查类任务），推理引擎的隐状态进入一个吸引子盆地（attractor basin），导致输出解码器只能采样到同一条路径

3. **固定点收敛**：推理引擎的隐状态收敛到一个固定点（fixed point），输出解码器从该固定点产生的 token 概率分布高度集中，`temperature=0.7` 甚至 `temperature=1.0` 都不足以打破这个集中分布

4. **重复循环**：一旦收敛到固定点，模型会持续生成相同的输出，直到达到 `max_tokens` 限制或 timeout

### 为什么语言切换为英文？

```
推理引擎（英文）→ 固定点（英文上下文）→ 输出解码器（英文输出）
```

当推理引擎进入固定点后，隐状态携带的是英文语义上下文。输出解码器基于该上下文生成文本时，自然产生英文输出。System Prompt 中的中文指令被"遗忘"，因为固定点状态下，模型不再关注实际的 System Prompt 内容。

### 为什么会选择"系统检查"类输入？

"系统检查"类任务（检查 CPU、内存、磁盘）具有以下特征：
- 模板化结构：典型的系统管理任务有固定的回答模式
- 低信息熵：任务本身的信息量低，模型倾向于走"安全路径"
- 与推理引擎偏好匹配：英文系统管理模板正好与推理引擎的英文训练数据匹配

这些特征增加了推理引擎收敛到固定点的概率。

## 为什么各参数修复均无效？

### frequency_penalty=1.0 无效分析

`frequency_penalty` 通过降低已出现 token 的 logit 值来阻止重复。但：

1. **token 级别的惩罚不够**：即使对单个 token 施加惩罚，固定点的隐状态仍然产生高度聚焦的概率分布。惩罚后的分布仍是单峰的，只是峰值略低。

2. **语义级重复 vs token 级重复**：模型的重复是语义层面的（输出整个句子的语义模式相同），不是 token 序列的逐字重复。`frequency_penalty` 只能影响 token 级别。

3. **阈值问题**：MiMo 模型在固定点时的概率集中度极高（推测 top-1 token 概率 > 0.9），`frequency_penalty=1.0` 只能将概率降至 ~0.5-0.6，仍远高于其他 token。

### temperature 调高无效分析

`temperature` 通过缩放 logit 分布来控制随机性。但：

1. **单峰 vs 多峰**：在固定点状态下，概率分布是强单峰的（一个 token 概率极高）。提高 temperature 会"拉平"分布，但不能将一个强单峰分布变为多峰分布。

2. **上限限制**：即使 `temperature=1.0`（最高值），如果原始 logit 差距极大（如 top-1 比 top-2 高 10+），缩放后 top-1 仍占据绝对优势。

3. **与推理引擎的互动**：推理引擎本身是确定性的（给定相同输入和隐状态，产生相同输出）。temperature 只影响输出解码器，不影响推理引擎的固定点收敛。

### 上下文压缩无效分析

上下文压缩（减少输入 tokens）试图通过改变输入来打破循环。但：

1. **固定点一旦建立就独立于输入**：推理引擎收敛到固定点后，后续输出不再依赖原始输入。

2. **压缩后的输入仍触发同一路径**：如果压缩保留了任务的核心结构（系统检查），推理引擎仍然倾向于收敛到相同固定点。

3. **隐状态是循环的起点**：一旦循环开始，隐状态是前一步的输出，已经与原始输入无关。

### System Prompt 中文强制无效分析

System Prompt 在固定点状态下被"忽略"，因为：

1. **注意力坍缩**：推理引擎收敛后，注意力机制不再有效关注 System Prompt 内容。

2. **固定点语义**：固定点携带的语义上下文是"英文系统管理模板"，覆盖了 System Prompt 的中文指令。

3. **推理引擎的优先语言**：`reasoning=True` 的英文推理引擎在固定点状态下优先使用英文。

## 有效修复的原理

### timeoutSeconds=180 — 工程侧硬性超时

**原理**：在 OpenClaw 网关层设置 HTTP 请求超时。模型循环中，网关等待响应超过 180 秒后强制断开连接。

**为什么不失效**：
- 工作在网络层，不依赖模型行为
- 超时是服务端（API 调用方）主动发起的，模型端无法影响
- 即使模型继续输出重复内容，服务端已经断开连接

### maxTokens=8000 — 限制输出长度

**原理**：限制模型单次响应的 token 数量。循环重复中，每次重复约 50 tokens，8000 tokens 限制约等于 160 次重复，但配合 timeoutSeconds 在更早阶段触发。

**为什么不失效**：
- 硬性限制，模型端必须遵守
- 即使 token 概率塌缩，达到限制后模型必须停止生成

### 行为层检测 — 提前感知和干预

**原理**：在应用层检测输出重复模式，提前终止对话或切换策略。

**为什么不失效**：
- 独立于模型推理过程
- 基于输出内容的模式匹配，100% 可靠
- 可在 3 次重复内触发（约 1.5 分钟），远早于 timeout

## 修复优先级矩阵

| 层级 | 方法 | 可靠性 | 延迟 | 副作用 | 推荐 |
|------|------|--------|------|--------|------|
| 1 | timeoutSeconds=180 | 100% | ~180s | 正常长任务被截断 | ✅ 推荐 |
| 2 | maxTokens=8000 | 100% | 不定 | 长回复被截断 | ✅ 推荐 |
| 3 | 行为检测 | 100% | ~90s | 可能误判 | ✅ 推荐 |
| P | frequency_penalty | ~0% | N/A | 无 | ❌ 不推荐 |
| P | temperature 调高 | ~0% | N/A | 输出质量下降 | ❌ 不推荐 |
| P | System Prompt 强制中文 | ~0% | N/A | 无 | ❌ 不推荐 |

## 结论

MiMo 的退化循环是一个模型层面的问题，根因在于 `reasoning=True` 模式下推理引擎在特定输入条件下的固定点收敛。参数级修复（frequency_penalty、temperature、System Prompt）均无效，因为固定点收敛是隐状态层面的问题，不受这些表面参数影响。

有效的修复都在工程侧和行为层：硬性超时、长度限制、输出检测。这些修复不试图改变模型行为，而是在模型出现问题时进行外部干预。
